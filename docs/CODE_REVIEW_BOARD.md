#### #1-1-A: ペース補正の適用タイミングとフェーズ戻り時の中間スコア表示

##### 1. ユーザー視点: ツールがどう振る舞うべきか

ユーザー（GM）が途中修正のためにScene 1に戻り、修正後に再びフェーズを進める運用では、以下の挙動が期待される:

- **正常フロー（前進方向）:** 序盤→ペース→中盤→終盤と進む際、ペース補正はペースフェーズ通過後に1回だけスコアに加算される。**→ 現在の実装で正しく動作する。**
- **途中修正フロー（後戻り＋再前進）:** 例えば中盤1完了後に脚質を修正した場合、GMは修正後にScene 1から再度フェーズを進める。このとき:
  - ペースダイスの**結果（1d9の出目）は破棄されず保持**される（仕様: `scene1-setup.md` §4「影響のないデータの保護」）
  - ペース補正値は**変更後の脚質に基づいて再計算**される（仕様: `basic-rules.md` §6 Case 3）
  - GMが意識するのは「中盤1完了時点の正しいスコア」以降であり、それ以前のフェーズの中間スコアは通常参照しない
- **実害の可能性:** 途中修正後にScene 3を再通過する際、Startフェーズの表示段階でペース補正込みのスコアが一瞬表示される可能性がある。ただしGMはStartの中間スコアに基づいて判断を行わないため、**運用上の実害はほぼない**。

##### 2. コード実装: 現状と改善方針

- **現状:** `calculateTotalScore(participant, strategies, paceRoll)` はフェーズ認識を持たず、`paceRoll !== null` であれば常に補正を加算する。
- **改善案:** 引数に `currentPhaseId` を追加し、ペースフェーズ以降（Mid/End）のときのみ補正を加算する:
  ```ts
  // 現在
  if (paceRoll !== null) { total += mod; }
  // 改善後
  if (paceRoll !== null && isPostPacePhase(currentPhaseId)) { total += mod; }
  ```
- **ペースダイスの結果保持:** `useRaceStore.ts` の `paceResult` はScene 1に戻っても `resetRace()` を呼ばない限り保持されるため、この点は仕様通り。

##### 3. 仕様ドキュメント: Gap分析

- `basic-rules.md` §3 はペース補正の「前進方向の適用タイミング」のみを定義しており、**フェーズ後退時にスコア表示がどうあるべきか**は未定義。
- `scene1-setup.md` §4 は「設定修正とデータ維持」を定義しているが、**途中修正中の各フェーズの中間スコア表示ルール**（どのフェーズまでの補正を含めるか）は明記されていない。
- `basic-rules.md` §6 Case 3 は「ペース補正は維持された出目に基づき再計算」と述べているが、**再計算結果をスコアに反映するタイミング**（即座に？中盤フェーズ到達時？）が不明確。
- **推奨:** `basic-rules.md` §3 または §6 に「途中修正時のペース補正の再適用タイミング」を追記する。具体的には「ペース補正は常に中盤フェーズ以降の計算にのみ反映され、Startフェーズの表示にはいかなる場合も含まない」と明記すると、実装側の判断基準が明確になる。

#### #1-1-B: 着差判定ダイスの格納・参照パターンの脆弱性

##### 1. ユーザー視点: ツールがどう振る舞うべきか

レース結果画面で、各出走者間の「着差」（ハナ、アタマ、クビ、1/2バ身 等）が正確に表示されることが期待される。特に以下の複合シナリオで正確性が求められる:

- **シナリオ①: 3グループ連鎖 1点差** — A[100] > B[99] > C[98]
  - A vs B、B vs C の2つの着差判定（各 `1d2`）が独立して行われ、それぞれ正しい着差表記が表示される。
- **シナリオ②: 同点グループ + 1点差** — A1[100], A2[100] > B[99]
  - A1 vs A2 の写真判定（`1d5`）で順位確定（ハナ差）。その後、Aグループ vs B の着差判定（`1d2`）で着差表記が決定される。
  - **写真判定でA1とA2の順位が入れ替わったとしても、Aグループ全体とBの着差（アタマ/クビ）は影響を受けない。** 着差は「グループ間のスコア差」に基づく概念であり、グループ内の個人順位とは独立している。

##### 2. コード実装: 現状の問題と改善方針

**現状のデータ構造と参照パターン:**

```ts
// types/index.ts:62-65 — 出走者個人に1つずつ格納
judgment?: {
    photo?: number;   // 1d5 結果
    margin?: number;  // 1d2 結果
};
```

```ts
// RankingCalculator.ts:152 — スコア一致 + margin存在で間接検索
const upperRep = participants.find(
    p => p.score === prev.score && p.judgment?.margin !== undefined
);
```

**この設計が機能する条件（暗黙の前提）:**
- `margin` は `detectJudgmentNeeds` が選出した**上位グループの代表者（最小枠順）にのみ**格納される
- 各スコアレベルにつき、`margin` を持つ参加者は**最大1名**
- UI層が `detectJudgmentNeeds` の `representativeId` に正確に従って結果を格納する

**これらの前提が満たされている限り、基本的なシナリオでは正しく動作する。** 実際にトレースすると:

- **シナリオ①（A[100]>B[99]>C[98]）:** A vs B の margin は A に格納（score=100）、B vs C の margin は B に格納（score=99）。参照時、`prev.score` でそれぞれ正しい格納先を特定できる。✅
- **シナリオ②（A1[100],A2[100]>B[99]）:** margin は A1（代表者）に格納。B を処理する際、`prev.score === 100` で A1 を発見。A2 は margin を持たないためスキップされる。✅

**しかし、以下の設計脆弱性が存在する:**

1. **暗黙の結合:** `detectJudgmentNeeds`（代表者選出）と `calculateFinalRanking`（結果参照）が、「代表者のみが margin を持つ」という**コード上に明示されていない契約**で結合されている。UI層がこの契約を誤って破る（例: グループ全員に margin を格納する）と、`find()` が意図しない参加者を返す。
2. **境界の識別情報がない:** `Umamusume.judgment.margin` は単なる数値であり、**どの境界（どのスコアペア間）の判定結果なのか**を示す情報を持たない。将来的に判定ロジックが複雑化した場合（例: ハウスルールで着差判定条件が変わる等）、参照の正確性を保証できなくなる。
3. **`find()` の順序依存:** `participants` 配列の並び順に依存して結果が変わりうる。ソート前の元配列の順序はエントリー順であり、意図的に制御されていない。

**改善案: 境界単位でのデータ格納**

```ts
// Option A: RaceState に境界結果マップを追加
marginResults: Map<string, number>;  // key: "100-99", value: 1d2結果

// Option B: JudgmentRequest に境界IDを持たせ、結果と紐付け
interface JudgmentResult {
    boundaryKey: string;    // "100-99"
    type: 'photo' | 'margin';
    diceResult: number;
}
```

これにより、`calculateFinalRanking` では `scoreDiff === 1` の場合に境界キー `${prev.score}-${current.score}` で直接結果を引けるようになり、暗黙の前提に依存しなくなる。

##### 3. 仕様ドキュメント: Gap分析

- `scoring-and-judgment.md` §2 は着差判定の**アルゴリズム**（いつ、誰が、何を振るか）を定義しているが、**判定結果の格納と参照のデータモデル**（どこに、どのような構造で保持するか）は定義していない。
- 同 §3 は最終確定ロジックを定義しているが、「着差表記はグループ間の概念であり、写真判定による個人順位変動の影響を受けない」ことが**暗黙の前提として扱われており、明文化されていない**。
- **推奨①:** `scoring-and-judgment.md` §3 に「着差表記はスコアグループ間の差分に基づくものであり、同点グループ内の写真判定結果による個人順位の変動は着差表記に影響しない」と明記する。
- **推奨②:** `specs/architecture/` 配下、または `scoring-and-judgment.md` に判定結果のデータモデル（境界単位での格納方針）を追記し、実装側の設計指針とする。
